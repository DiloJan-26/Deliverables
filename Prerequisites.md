Needed to publish on the Play Store. Also where subscriptions will live *in the next version* (not v0).

### 5.1 Google Play Console Account
| | |
|---|---|
| **What it is** | The account to publish Android apps on the Play Store |
| **Status** | ❓ **CHECK IF WE HAVE ONE** — needed for launch |
| **Cost** | 🔵 **ONE-TIME $25** (~₹2,100), lifetime, covers unlimited apps |
| **Who** | 👤 **Founder should own the account** (it's the company's developer identity), but add me as a user |

**Why founder owns it:** the Play Console account is tied to the company's legal/payout identity — bank details, tax info, the developer agreement. That's founder/company territory. But I need **admin or release access** to upload builds and configure technical settings.

**Inch-by-inch (founder creates, then adds dev):**
1. Founder goes to **play.google.com/console**.
2. Sign in with the **company Google account** (not a personal one — this becomes our developer identity).
3. Pay the **$25 one-time** registration fee.
4. Complete the identity verification (name, address, sometimes ID — this is why founder does it).
5. Once set up: **Users and permissions → Invite new users** → add the backend dev's email with **Admin** (or at least release + app-access) permissions.

> **Important for founder — start this FIRST (longest lead time):** Google requires identity verification for new developer accounts. There are two account types:
> - **Personal account** ($25, faster) — needs a government ID + a real Android device for verification, and requires a 12-tester closed test for ~14 days before public launch.
> - **Organization/Company account** ($25, more credible for a company) — **requires a D-U-N-S number** (a free business identifier from Dun & Bradstreet that can take **1–4 weeks** to obtain if we don't have one).
>
> **Recommendation:** If we want the app published under the company name (more professional, better for a real product), choose **Organization** — but **request the D-U-N-S number immediately**, because it's the single longest-lead item in this whole list. If we want to launch fastest and switch later, a Personal account works for v0. This is a founder/business decision — happy to advise, but the legal identity choice is yours.
>
> **Payment note:** the $25 must be paid with a **regular credit/debit card — prepaid and most virtual cards are rejected.** Verification typically completes in a few hours to 2 business days (excluding the D-U-N-S wait).

# 🔧 What the DEV should own directly (recommendation)

You asked whether you should handle some items instead of routing through the founder. **Yes — request *account access* from the founder, then do these yourself.** Here's why each is better in your hands:

| Item | Why you should own it |
|---|---|
| **Firebase Admin SDK key** | It's a master secret. Generating and storing it yourself (not emailed around) is basic security hygiene. |
| **Firestore setup + security rules** | Technical config; a wrong setting here exposes user data. Dev judgment needed. |
| **Lock auth to Google** | One setting, but security-relevant. |
| **Play Integrity enable** | Technical; tied to the abuse system you're building. |
| **AI key management + budget alerts** | You understand the cost model; you should watch the spend and set alerts. |
| **Hosting config + deploy** | You own the deployment pipeline. |
| **All env vars / secrets** | Secrets should live with the person who understands them, in a vault — not in a founder's inbox. |

**The clean division:** Founder owns *accounts and money* (Play Console identity, billing approval). You own *technical setup and secrets*. You ask the founder only for **access**, not for them to configure technical things they don't understand. This is both safer and faster.

**Concrete ask to your founder:** *"Please (1) create the Play Console account and add me as Admin, and (2) add my email as an Owner on the Firebase/Google Cloud project. Once I have access, I'll handle all the technical setup myself."*
